Voraussetzungen

ToolVersionZweck
Docker≥ 24Container-Runtime
Docker Compose≥ 2.20Multi-Service-Orchestrierung
GitbeliebigVersionskontrolle
Python 3.12 (optional)3.12+Nur für lokale Backend-Entwicklung ohne Docker

GPU-Hinweis: PyTorch wird mit einem CPU-only-Wheel installiert. GPUs mit sm_61 (z.B. GTX 1070) sind mit PyTorch ≥ 2.1 inkompatibel. Das ist eine bewusste Designentscheidung.

Schnellstart

# Repository klonen
git clone <repo-url>
cd Töltonaut

# Alle Services starten (erster Start lädt Images + baut Backend)
docker compose up

# Im Hintergrund starten
docker compose up -d

# Logs beobachten
docker compose logs -f backend

# Services stoppen
docker compose down

Nach dem Start:

  • Frontend: http://localhost:5173
  • Backend API: http://localhost:8000
  • Swagger UI: http://localhost:8000/docs
  • ReDoc: http://localhost:8000/redoc

Rebuild nach Code-Änderungen

# Backend-Image neu bauen
docker compose up --build backend

# Nur Backend-Container neu starten
docker compose restart backend

Lokale Entwicklung (ohne Docker)

Backend

cd backend

# Virtuelle Umgebung
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate

# PyTorch CPU zuerst (wichtig!)
pip install torch torchvision --index-url https://download.pytorch.org/whl/cpu

# Restliche Abhängigkeiten
pip install -r requirements.txt

# Umgebungsvariablen setzen
export DATABASE_URL="postgresql+psycopg://toeltonaut:toeltonaut@localhost:5432/toeltonaut"
export SECRET_KEY="dev-secret-key-change-in-prod"

# Server starten (Hot-Reload)
uvicorn app.main:app --reload --port 8000

Frontend

cd frontend
npm install
npm run dev        # Dev-Server auf Port 5173
npm run build      # Produktions-Build nach frontend/dist/
npm run preview    # Produktions-Build lokal vorschauen

API-Referenz: Frames & Keypoints

Frame als JPEG

MethodePfad
GET/api/frame/{job_id}/{frame_nr}

Gibt den Frame als image/jpeg zurück. Kann direkt in <img src="..."> verwendet werden.

Keypoints für einen Frame

MethodePfad
GET/api/keypoints/{job_id}/{frame_nr}

Optionaler Query-Parameter ?ms=<millisekunden>: Zeitbasierter Frame-Lookup – gibt Keypoints des Frames zurück, dessen timestamp_ms dem angegebenen Wert am nächsten liegt.

Annotation speichern

MethodePfad
POST/api/annotations/{job_id}/{frame_nr}
{
  "keypoints": [
    { "name": "l_front_hoof", "x": 0.245, "y": 0.918, "confidence": 1.0 },
    ...
  ]
}

API-Referenz: Authentifizierung

MethodePfadBeschreibung
POST /api/auth/register Registrierung mit E-Mail + Passwort (min. 8 Zeichen)
POST /api/auth/login Login (OAuth2 Password Flow) → JWT-Token
GET /api/auth/me Aktuellen User abrufen (Bearer-Token erforderlich)
DELETE /api/auth/account Account DSGVO-konform löschen (alle Daten, CASCADE)

Token-Gültigkeit: 24 Stunden. Das Frontend speichert den Token in localStorage (frontend/src/auth.ts).

API-Referenz: Training, Lern-Loop & ML

MethodePfadBeschreibung
GET /api/learning-status Lernstatus: Modellversion, Gesamtvideos, Trainingsvideos, Frames, annotierte Frames, Gangart-Verteilung
GET /api/export/{job_id} JSONL-Export aller Keypoints eines Jobs (ein JSON-Objekt pro Zeile: frame_nr, timestamp_ms, keypoints)
GET /api/export-coco/{job_id} COCO ZIP-Export: coco.json + Frames für MMPose Fine-tuning
DELETE /api/annotations/{job_id} Setzt alle manuellen Annotationen für einen Job zurück (Keypoints bleiben erhalten)
GET /api/training/export/{job_id} COCO-JSON für MMPose-Training herunterladen (einzelnes Video)
GET /api/training/export-bulk Bulk-COCO-ZIP aller Trainingsvideos mit Freigabe (coco.json + Frame-JPEGs)
GET /api/training/jobs Training-Jobs auflisten
POST /api/training/jobs Neuen Training-Job manuell anlegen
POST /api/training/start Fine-tuning starten: Bulk-ZIP aus DB erzeugen, TrainingJob anlegen, Hintergrund-Thread starten → 202
GET /api/training/status/{job_id} Echtzeit-Fortschritt (Epoche, Loss, Status); DB-Fallback nach Container-Neustart
POST /api/training/activate/{job_id} Modell aktivieren: config.yaml schreiben, Singletons invalidieren; liest output_model_path aus DB wenn In-Memory-State fehlt (Rollback)
POST /api/job/{job_id}/reanalyse Gangart-Erkennung auf vorhandenen DB-Keypoints neu ausführen – kein Re-Upload nötig; aktualisiert frames.gait und videos.gait_detected

Backend-Module

video_processor.py – VideoProcessor

Herzstück der Analysepipeline. Läuft in einem threading.Thread (CPU-bound, daher kein asyncio).

class VideoProcessor:
    def process(input_path, output_path, progress_callback, video_db_id=None) → str
    # Rückgabe: dominant erkannte Gangart als String

Intern: _DbWriter sammelt Frames + Keypoints und schreibt sie in Batches à 100 Einträge über psycopg3 COPY.

gait_detector.py – GaitDetector

class GaitDetector:
    def update(keypoints: list[tuple[int,int]], bbox: tuple[int,int,int,int], is_side_view: bool = True) → None
    def detect() → GaitResult  # { name, confidence, note }

Arbeitet auf einem gleitenden 30-Frame-Fenster. Analysiert normalisierte Y-Positionen der Fesselgelenke (Horse-10-Indizes: l_front=3, r_front=6, l_hind=14, r_hind=19, Sprunggelenke l_hock=13, r_hock=18). Nur is_side_view=True-Frames fließen in die LAP/DF-Berechnung ein.

tolt_scorer.py – ToltScorer

class ToltScorer:
    def score(tracks: dict[str, list[TaktTrackPoint]], fps=25.0) → ToltScore
    # ToltScore: score, feif_grade, errors, takt_regularity, lap, df, subclassification

trainer.py – Fine-tuning Pipeline

def run_finetuning(job_id, zip_path, base_checkpoint, base_config, output_model_path) → dict
def start_training_thread(job_id, zip_path, ..., db_url) → None  # Daemon-Thread-Einstiegspunkt

class HorseKeypointDataset  # liest Bulk-COCO-ZIP, erzeugt 64×64-Gaussian-Heatmaps
class TrainingState          # job_id, status, epoch, total_epochs, loss, output_model_path

Strategie: Backbone eingefroren, nur model.head (~726 Parameter) trainiert. CPU-realistisch in 30–60 Min für ~300 Frames / 50 Epochen. MLflow file-based (/app/mlruns), kein Server nötig. output_model_path wird in dataset_snapshot (JSONB) gespeichert – Rollback nach Neustart ohne erneutes Training möglich.

pose_factory.py – PoseFactory

def create_estimator(ai_config: AIConfig) → BasePoseEstimator:
    if ai_config.model_version == "v0.2-mmpose":
        try: return MMPosePoseEstimator(...)
        except ImportError: return PropPoseEstimator(...)
    return PropPoseEstimator(...)

AI-Konfiguration wechseln

Modell-Version in backend/config.yaml setzen, dann Backend neu starten:

# v0.1 – Proportionale Keypoints (Standard, kein zusätzliches Setup)
ai:
  model_version: "v0.1"
  model_path: "yolov8n.pt"

# v0.2 – MMPose hrnet_w32_horse10 (22 Keypoints, mAP ~88%)
ai:
  model_version: "v0.2-mmpose"
  model_path: "yolov8n.pt"
  pose_model: "hrnet_w32_horse10"
  pose_config: "/path/to/td-hm_hrnet-w32_8xb64-100e_horse10-256x256.py"
  device: "cpu"
MMPose Setup: Muss manuell im Backend-Container installiert werden:
pip install mmengine mmcv mmdet mmpose
Außerdem müssen Modell-Checkpoint und Config-Datei vorhanden sein.

Neue Gangart hinzufügen

Beispiel: Trab-Scoring hinzufügen.

  1. Neue Datei backend/app/trab_scorer.py nach dem Muster von tolt_scorer.py erstellen
  2. In video_processor.py: TrabScorer importieren und in der Pipeline aufrufen
  3. In models.py: Pydantic-Response-Schema TrabScoreResponse hinzufügen
  4. In main.py: GET-Endpoint /api/trab-score/{job_id} registrieren
  5. In frontend/src/api/client.ts: getTrabScore() hinzufügen
  6. Frontend-Komponente TrabScoreCard.tsx erstellen
  7. i18n-Strings in locales/de.json und en.json ergänzen

State-Management

Kein Redux oder Zustand-Library – State läuft in App.tsx per useState/useEffect und wird als Props nach unten gereicht. Auth-Token in localStorage via auth.ts.

API-Client (client.ts)

Alle Backend-Anfragen laufen durch frontend/src/api/client.ts. Upload verwendet XHR statt fetch für den Fortschrittsbalken:

// Beispiel: Video hochladen
import { uploadVideo } from './api/client'

const { job_id } = await uploadVideo(file, {
  horse_name: 'Freyja',
  gait_label: 'Tölt',
  camera_angle: 'seitlich_links'
}, (progress) => setProgress(progress))

// Beispiel: Analyse abrufen
const score = await getToltScore(job_id)

Modell upgraden (v0.1 → v0.2)

  1. MMPose installieren (s. AI-Konfiguration)
  2. Modell-Checkpoint hrnet_w32_horse10.pth herunterladen: 
    mim download mmpose --config td-hm_hrnet-w32_8xb64-100e_horse10-256x256 --dest .
  3. config.yaml aktualisieren (s. oben)
  4. Backend neu starten: docker compose restart backend
  5. Health-Check: curl localhost:8000/health

COCO-Export für Training

# Export über API
curl -o training_data.json \
     http://localhost:8000/api/training/export/{job_id}

# MMPose Training starten (Beispiel)
python tools/train.py \
  configs/body_2d_keypoint/topdown_heatmap/horse10/... \
  --cfg-options \
    data.train.ann_file=training_data.json

Das exportierte JSON folgt dem COCO-Keypoint-Format mit 17 Keypoints (v0.1) bzw. 22 (v0.2). Nur Videos mit gesetztem training_consent=true werden exportiert.
